Q4 · Sequence Analysis

При 56 лейблах 71% токенов — schema

В uni-encoder архитектуре schema tokens и text tokens кодируются вместе. С ростом лейблов schema начинает доминировать в последовательности.

6 labels — schema 46%

Schema: 64 tokens
Text: 75 tokens

26 labels — schema 66%

Schema: 148 tokens
Text: 75 tokens

56 labels — schema 71%

Schema: 185 tokens
Text: 75 tokens
→
Bi-encoder opportunity: при кэшировании 185 schema-токенов per-request sequence сокращается до ~75 токенов. Теоретический speedup: ~2.2x (attention O(n²)).
Medium text: 62 words, 399 chars DeBERTa max_position_embeddings = 512

Conclusions

Выводы и рекомендации

Подтверждённые гипотезы

H9
56 лейблов не деградируют safety F1 (+0.045)
H8
Больше лейблов → больше latency (+20% max)
H5
Оптимальный batch: 16–32 (interactive), 64 (bulk)

Ограничения

!
BiEncoder: upstream arch mismatch, excluded
!
XSTest labels: auto-split, requires manual review

Production рекомендации

1
Использовать full schema (56 labels) — нет penalty по quality
2
SchemaRegistry — plugin-first, dynamic label management
3
Sequence guard — auto-truncate при >512 tokens
4
Bi-encoder caching — future work, ~2.2x potential
